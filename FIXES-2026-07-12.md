## P1 — BUG-A · Réponses SIMULÉES présentées comme réelles (confiance)

**Gravité : critique.** L'app fabrique une réponse (texte générique de `REPLIES`) et l'affiche comme une vraie réponse, 0 token, sans le dire clairement à l'utilisateur.

**Reproduction (vérifiée en live) :**
1. Sélectionner un modèle image (FLUX.2 Pro) → il devient `currentModel` et est **persisté en localStorage** (`syncModel` → `DB.set('model', currentModel)`, ~L1917).
2. Recharger la page. Au boot, `currentModel` est restauré depuis la DB (**= FLUX**) mais `currentMode` démarre à `'text'` (~L2909). → **désync mode/modèle**.
3. Envoyer un message texte normal. `runGen` détecte un modèle qui n'est ni OpenRouter ni SiliconFlow-texte → tombe dans le repli simulé.

Constaté à l'écran : réponse « *Pour l'instant je simule — mais l'interface est 100 % fonctionnelle ✦* », et pire, un composant React sans rapport avec la question (« ma ratatouille » → code `Button`). Titre auto = « Simulation d'assistant ».

**Cause racine :**
- `~L1656-1657` (boot) : `currentModel` restauré sans vérifier que son `kind` est compatible avec le mode texte de démarrage.
- `~L2440-2441` (runGen) : `if(!isOR && !isSF){ return finish(REPLIES[ri++%REPLIES.length]); }` — le repli simulé était pensé pour « pas de clé », il attrape aussi « modèle média en conversation texte ».

**Correctif proposé (defense-in-depth, les deux) :**

1. Au boot, forcer un modèle texte si le modèle persisté est un modèle média (racine) :
```js
// après :  if(!MODELS.some(m=>m.id===currentModel))currentModel=MODELS[0].id;
{ const _m=modelById(currentModel); if(_m && /image|video|audio/.test(_m.kind||'')) currentModel='MAZ Auto'; }
```
2. Garde dans `runGen` — si un modèle média se retrouve en conversation texte, router vers l'assistant au lieu de simuler :
```js
if(!isOR && !isSF){
  if(MODE_CAT[currentMode]==='text'){ opts.model = currentModel = 'MAZ Auto'; syncModel(); return runGen(node,_,opts); } // pas de boucle : currentModel change
  return finish(REPLIES[ri++%REPLIES.length]); // vrai cas "sans clé"
}
```
Vérifier que `modelById`, `MODE_CAT`, `syncModel` sont accessibles à ces endroits (functions hoistées / const définies plus haut).

**Test de non-régression :**
- En console : `localStorage.setItem('mzai:model', JSON.stringify('FLUX.2 Pro'))` puis reload → le composer doit afficher **MAZ Auto**, pas FLUX.
- Envoyer « bonjour » → réponse **réelle** (ligne de stats avec tokens > 0, un POST `openrouter.ai/.../chat/completions` dans l'onglet réseau), aucune mention « je simule ».
- Vérifier que le vrai cas « sans clé » (vider les clés dans Réglages) affiche toujours le repli simulé proprement.

## P1 — BUG-B · Mode Traduction/one-shot traduit tout l'historique

**Gravité : élevée** (résultat faux, silencieux).

**Reproduction (vérifiée) :** dans une conversation contenant des prompts de génération média/voix (messages `user` porteurs de `.text` mais **sans réponse `ai` texte**), passer en mode Traduction et traduire une phrase neuve → la sortie traduit **aussi les prompts précédents**. Constaté : image → voix → traduction a renvoyé les 3 phrases traduites en anglais.
(En dialogue texte classique user↔ai le modèle ne traduit que le dernier tour — le bug se manifeste dès qu'il y a des messages user « orphelins ».)

**Cause racine :** `~L2447-2467` — `runGen` construit `messages` à partir de **tout** `c.msgs` (`allMsgs = c.msgs.filter(m=>m.text)`, fenêtre 24). Or Traduction / Transcription sont des tâches **one-shot** : elles ne devraient recevoir que le dernier message utilisateur. Le prompt système dédié (L2368) ne suffit pas à contrer un historique de plusieurs phrases sources.

**Correctif proposé :** pour les modes one-shot, n'envoyer que le dernier message user.
```js
// juste avant la construction de `messages` (~L2455)
const ONE_SHOT = (currentMode==='translate' || currentMode==='transcribe');
const histMsgs = ONE_SHOT ? trimmedMsgs.slice(-1) : trimmedMsgs;
// puis .concat(histMsgs.map(...)) au lieu de trimmedMsgs.map(...)
```
(Garder Doc et Code sur l'historique complet — ils en ont besoin.)

**Test de non-régression :**
- Nouvelle convo → un échange texte normal (« retiens 4271 ») → passer en Traduction FR→EN → traduire « Le chat dort sur le canapé. » : sortie = **uniquement** « The cat is sleeping on the couch. », sans reprise du 4271.
- Refaire image → voix → traduction : seule la phrase saisie est traduite.

## P2 — BUG-C · LaTeX affiché brut

**Gravité : moyenne** (cosmétique mais visible sur tout contenu mathématique — DeepSeek en produit spontanément).

**Reproduction (vérifiée) :** demander une formule → l'app affiche littéralement `\[ A = \pi r^2 \]`, `\frac{...}`, `\text{...}`. Aucun nœud KaTeX/MathJax dans le DOM.

**Cause racine :** `~L2225-2228` `renderMD()` ne gère ni `\[ … \]`, ni `\( … \)`, ni `$…$`. Le contenu est `esc()`-échappé puis laissé tel quel.

**Correctif proposé :** rendu maths léger, **sans dépendance externe** (CSP `connect-src`/`script-src` verrouillée — pas de CDN KaTeX possible). Deux options :
- **Option légère (recommandée)** : convertir les délimiteurs en un rendu HTML/CSS basique (italique + exposants `<sup>`, fractions en `a⁄b`). Suffit pour 90 % des cas (formules inline courtes). Post-traitement dans `renderMD` après `esc()`.
- **Option complète** : embarquer KaTeX **inline** (JS + CSS collés dans le fichier, police en data-URI). Coûte ~270 Ko au fichier et va à l'encontre du mono-fichier léger — à ne faire que si Mazen veut du rendu maths pro.

Décision Mazen requise sur l'option. Défaut = option légère.

**Test de non-régression :** « donne l'aire d'un cercle en LaTeX display » → plus de `\[` ni `\pi` bruts à l'écran ; le texte normal et les blocs de code restent intacts.

## P2 — BUG-D · Génération liée à la vue, pas à la conversation (ARCH-03)

**Gravité : moyenne. Pas de perte de données** (vérifié : le média/texte est bien écrit dans le store et réapparaît en re-naviguant).

**Deux symptômes, une cause :**
1. **Composer verrouillé partout** : pendant une génération, `setGenerating(true)` agit sur l'UI **globale** (~L2640). Changer de conversation laisse le bouton Stop visible et le composer bloqué dans la nouvelle convo, alors qu'on n'y génère rien.
2. **Média non re-rendu au retour** : si la génération (image/vidéo Wan, ~197 s) se termine alors qu'on a changé de vue, `place()` écrit dans un `bubble` DOM détaché (la vue a été vidée par `chat.innerHTML=''` dans `openConvo`, ~L2812). Le média est bien `saveConvos()` → il **réapparaît en re-cliquant la conversation**, mais pas en direct. Vécu comme un échec.

**Cause racine :** l'état de génération (`curStop`, `setGenerating`, le `node`/`bubble` cible) est global et couplé à la vue active, pas à un id de conversation. C'est le chantier ARCH-03 déjà identifié — mais **moins lourd que craint** : les données transitent correctement, il manque le rattachement vue↔convo.

**Piste de correctif (à faire posément, pas en un patch) :**
- Indexer la génération en cours par `convoId` (map `generating[convoId] = {ctrl, node}`).
- `setGenerating` ne verrouille le composer que si `activeId === convoId` de la génération.
- À la fin de `place()`/`finish()`, si `activeId !== convoId`, ne pas toucher le DOM — juste `saveConvos()` + `renderRecents()` (déjà fait) ; le re-render se fera à l'ouverture.
- Optionnel : un discret badge « génération en cours » sur la ligne de la convo dans la sidebar.

**Test de non-régression :**
- Lancer une vidéo Wan, changer de conversation immédiatement : le composer de l'autre convo doit être **utilisable** (pas de Stop fantôme). Revenir sur la convo d'origine à la fin → la vidéo est là.
- Envoyer un texte dans une 2ᵉ convo pendant que la 1ʳᵉ génère : les deux aboutissent sans écrasement d'état.

## P2 — BUG-E · Médias générés : URLs distantes qui expirent (API-02)

**Gravité : moyenne** (perte visuelle différée — l'utilisateur croit avoir gardé l'image).

**Confirmé en live :** l'URL d'image FLUX (`delivery.us2.bfl.ai/…?se=2026-07-12T20:11:47Z&sig=…`) et la vidéo Wan (`…aliyuncs.com/…?Expires=1783973339&Signature=…`) sont **signées et expirent** (image ~1 h). Après expiration → image/vidéo morte dans l'historique.

**Cause racine :** `~L2646/2662` `place()` stocke `media.url` = l'URL fournisseur telle quelle dans `c.msgs`. Rien n'est rapatrié localement.

**Correctif proposé :** à la génération (dans `place()`, avant `saveConvos`), fetch le média et le stocker en **IndexedDB `maz-media`** (déjà en place pour les autres médias lourds via `MEDIA` + réf `_mid`), comme pour les images user. `mediaBubbleHTML` / `rehydrateFiles` lisent alors le blob local.
- ⚠️ Vérifier la **CSP `connect-src`** : les domaines de livraison (`delivery.*.bfl.ai`, `*.aliyuncs.com`, Kling) doivent être autorisés pour le fetch de rapatriement, sinon le fetch est bloqué. C'est ce qui faisait juger API-02 « impossible » en juin — c'est faisable en ajoutant précisément ces hôtes à `connect-src` (pas un `*`).
- Nécessite un vrai test avec crédits (génération réelle) avant livraison.

**Test de non-régression :** générer une image → recharger la page → l'image s'affiche toujours (servie depuis IndexedDB, pas depuis l'URL signée) ; l'export reste portable.

## Vérifiés SAINS (pas des bugs — ne pas « corriger »)

- **XSS** : `<img onerror>`, `<script>`, `[lien](javascript:…)` tous neutralisés en live (`esc()` + regex liens http/https-only L2228). Aucun exécuté, aucun `javascript:` dans le DOM.
- **Quota localStorage** (ARCH-02) : `DB.set` en try/catch avec toast « Stockage local plein » (L1521) ; historique corrompu → copie de secours `convos.bak` (L1536).
- **Erreur mid-stream SSE** (API-01) : `if(data.error){ streamErr=… }` remonte l'erreur (L2517).
- **AbortController** : présent pour le texte (timeout adaptatif 180 s/90 s, L2480-2481) et le média (`curStop`/`done`, L2644 — stoppe le polling).
- **API-08** : repli SF = `deepseek-ai/DeepSeek-V3` (L3326), plus le V2.5 obsolète.

## Point mineur noté

- **API-06** : le Stop coupe le **polling** média (`done=true`, boucle sort) mais n'aborte pas le fetch `/submit`/`/status` en vol. Impact facturation **négligeable** : une fois `/submit` envoyé, le job SF/Kling est facturé de toute façon. À ignorer sauf souci de propreté.
- **Modèle média persistant après sortie du mode** : lié à BUG-A. Le fix boot + garde runGen le neutralise ; vérifier au passage que « Nouvelle conversation » remet bien un modèle texte (sinon ajouter le reset dans le handler du bouton).
